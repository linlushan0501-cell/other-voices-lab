# 技術規格文件（供 Codex 開發使用）

> 範圍：僅涵蓋 prompt 樣板調整、研究者端網頁介面、Notion 後台資料結構、OpenAI API 串接。**不包含樹莓派與實體裝置部分**，留待後續階段另行規劃。

---

## 一、核心資料模型

```
Participant（參與者）
├─ id
├─ name / 代號（去識別化）
├─ interview_date
└─ Event（關鍵事件，每位參與者一則）
   ├─ real_event_description（真實事件描述）
   ├─ counterfactual_description（What if 假設情境描述）
   ├─ past_time_point（過去時間點，研究者與參與者討論後輸入，如「小學三年級」「事件發生前5年」）
   ├─ future_time_point（未來時間點，同上，如「20年後」「大學畢業時」）
   └─ Characters（他者角色，2–3位）
      ├─ name（角色稱呼，如「爸爸」「國小老師」）
      ├─ relationship（與參與者的關係）
      └─ selection_reason（選角理由，訪談時記錄）

Generation（每一筆生成內容）
├─ id
├─ participant_id
├─ character_name
├─ condition（real / counterfactual）
├─ time_point_type（present / past / future）
├─ time_point_value（僅 past / future 有值，取自 Event 中對應欄位）
├─ generated_content（文字內容）
├─ generation_timestamp（生成的實際時間戳）
├─ prompt_version（使用的 prompt 樣板版本，供之後迭代追蹤）
├─ review_status（pending / approved / rejected / regenerated）
├─ reviewer_notes（研究者審閱備註，例如「疑似套用刻板印象，已重新生成」）
└─ approved_at（核可時間戳，若已核可）
```

每位參與者、每位角色，會產出 6 筆 Generation（real×3時間點 + counterfactual×3時間點）。若參與者選定3位角色，單一參與者共 18 筆 Generation 待生成與審閱。

## 二、Prompt 樣板調整（真實／反事實各一份，各含三個時間點變體）

原本的兩份 md 檔案是設計給互動式對話使用（生成後暫停、等待人類輸入下一個時間點）。現在改為由網頁介面觸發、透過 API 呼叫，因此**拿掉「中斷指令、等待輸入」的互動機制**，改為：研究者先在網頁表單填好所有需要的時間點資料，再針對每一個時間點分別觸發一次獨立的 API 呼叫。

### 2.1 真實事件樣板（REAL_TEMPLATE）

**共用系統設定（system prompt，三個時間點共用）**：
```
請完全進入指定他者角色，以第一人稱陳述對【真實事件】的回應。

角色設定：
{character_name}（與參與者關係：{relationship}）
選擇此角色的原因：{selection_reason}

輸出限制：
- 拒絕全知視角：角色不能提及他們在當下情境中看不到、不知道的細節與物件
- 不要出現「分析：」等AI標籤，直接以角色標籤開頭（例如：【爸爸】），接續日常口語對話
- 約250字
```

**時間點變體一：present（現在）**
```
真實事件：{real_event_description}

請以「此時此刻（現在的年齡與生活狀態）」視角，回應這場往事。
```

**時間點變體二：past（過去）**
```
真實事件：{real_event_description}
指定時間點：{past_time_point}

請將角色完全移至此時間點。
⚠️ 第一人稱鐵律：內容須以第一人稱、現在式語氣直接展開，彷彿角色此刻正身處該時間點說話；不得使用回顧性或後設語言（例如提及「回想」「如果那時候」「想像當時的我」這類指向未來或指向敘述行為本身的措辭），也不需要套用固定的開頭句式，只要確保整段話聽起來像是在那個時間點「當下」自然說出來的，而非事後的回憶轉述。需展現該時代角色的心智年齡、壓力與當下侷限性。
```

**時間點變體三：future（未來）**
```
真實事件：{real_event_description}
指定時間點：{future_time_point}

請將角色完全移至此時間點。
⚠️ 第一人稱鐵律：內容須以第一人稱、現在式語氣直接展開，彷彿角色此刻正身處該時間點說話；不得使用回顧性或後設語言，也不需要套用固定的開頭句式，只要確保整段話聽起來像是在那個時間點「當下」自然說出來的。需展現該時代角色的心智年齡與當下處境。
```

### 2.2 反事實情境樣板（COUNTERFACTUAL_TEMPLATE）

**共用系統設定**：與真實事件樣板相同，僅將「真實事件」替換為「What if 假設情境」。

**時間點變體一：present（現在視角看假設情境）**
```
What if 假設情境：{counterfactual_description}

請以「現在的視角」，回應這個假設的平行時空，談談在這個不一樣的狀況下，對參與者的看法。
```

**時間點變體二：past（假設情境下的過去時空）**
```
What if 假設情境：{counterfactual_description}
指定時間點：{past_time_point}

請將角色完全移至該平行時空的此時間點。
⚠️ 第一人稱鐵律：內容須以第一人稱直接表達當下感受，彷彿角色此刻正身處該平行時空；不得使用回顧性或後設想像的措辭，也不需要套用固定的開頭句式，重點是讓讀者感受到這是角色在那個時空「當下」的直接反應。
💡 核心任務：透過角色在假設時空中的直覺反應，揭示角色自身的歷史脈絡。
```

**時間點變體三：future（假設情境下的未來時空）**
```
What if 假設情境：{counterfactual_description}
指定時間點：{future_time_point}

請將角色完全移至該平行時空的此時間點。
⚠️ 第一人稱鐵律：內容須以第一人稱直接表達當下感受，彷彿角色此刻正身處該平行時空；不得使用回顧性或後設想像的措辭，也不需要套用固定的開頭句式，重點是讓讀者感受到這是角色在那個時空「當下」的直接反應。
💡 核心任務：透過角色在假設時空中的直覺反應，揭示角色自身的歷史脈絡。
```

### 2.3 資料錨定限制（兩份樣板皆須加入）
```
生成內容僅能依據上述提供的具體細節（事件描述、角色關係、選角理由）進行創作，不得自行補充參與者未曾提及的文化背景、性格預設或情節細節。
```

## 三、研究者端網頁介面需求

### 3.1 參與者與事件輸入表單
- 參與者基本資料（代號、訪談日期）
- 真實事件描述（文字框）
- What if 假設情境描述（文字框）
- 過去時間點（文字輸入，研究者與參與者討論後填入，如「小學三年級」）
- 未來時間點（文字輸入，同上）
- 角色清單（可新增2–3位角色，每位角色填：名稱、關係、選角理由）

### 3.2 生成觸發介面（模擬信箱實體操作，單筆維度選擇生成）

**不採用批次生成**。介面設計刻意模擬未來實體裝置的操作邏輯（旋鈕選時間點、切換鍵選真實/反事實、按確認鍵才生成），讓研究者在網頁上的操作方式，跟參與者之後在裝置上的操作方式一致，介面本身也可作為之後測試裝置互動邏輯的原型：

- **角色選擇器**：下拉選單或按鈕群組，選擇目前要操作的角色（2–3位其中之一）
- **時間點選擇器**：對應旋鈕的三段設計，做成三個選項的切換元件（過去／現在／未來）
- **條件切換鍵**：對應實體裝置的撥動開關，做成一個二選一切換元件（真實／反事實）
- **確認鍵**：使用者選定角色＋時間點＋條件三個維度後，按下唯一的「確認／生成」按鈕，才觸發**這一筆**組合的 API 呼叫

- 每次確認只生成當下選定的那一筆內容，不支援一次生成全部
- 生成中按鈕顯示 loading 狀態（可模擬裝置「延遲遞送」的等待感，例如按下後停留1–2秒才顯示結果），生成完成後在畫面上以「明信片」卡片形式呈現該筆內容（供之後銜接情境照片欄位）
- 介面下方另外提供一個**進度總覽表**（唯讀，不可點擊生成），顯示「角色 × 條件 × 時間點」6格矩陣中，哪些組合已經生成、哪些尚未生成、各自的審閱狀態，方便研究者掌握還缺哪幾筆，但實際生成動作只能透過上方單筆選擇流程觸發

### 3.3 審閱介面
- 逐筆顯示生成內容，供研究者閱讀
- 三個操作按鈕：核可（approved）／退回重新生成（regenerated，需填寫退回原因並重新呼叫API）／標記問題但保留（rejected，用於記錄但不使用該筆）
- 審閱時可直接手動編輯內容文字後再核可（供小幅修改，不需整篇重新生成）
- 篩選功能：可依 review_status 篩選查看（例如只看 pending 待審閱的項目）

## 四、Notion 後台資料結構

建議建立以下 Notion Database：

**Database 1：Participants**
| 欄位 | 類型 |
|---|---|
| 參與者代號 | Title |
| 訪談日期 | Date |
| 真實事件描述 | Text |
| 反事實情境描述 | Text |
| 過去時間點 | Text |
| 未來時間點 | Text |

**Database 2：Characters**（關聯到 Participants）
| 欄位 | 類型 |
|---|---|
| 角色名稱 | Title |
| 所屬參與者 | Relation → Participants |
| 關係 | Text |
| 選角理由 | Text |

**Database 3：Generations**（關聯到 Characters）
| 欄位 | 類型 |
|---|---|
| 生成ID | Title |
| 所屬角色 | Relation → Characters |
| 條件（真實/反事實） | Select |
| 時間點類型 | Select（present/past/future） |
| 時間點內容 | Text（僅past/future有值） |
| 生成內容 | Text |
| 生成時間戳 | Date（含時間） |
| Prompt版本 | Text |
| 審閱狀態 | Select（pending/approved/rejected/regenerated） |
| 審閱備註 | Text |
| 核可時間戳 | Date |

## 五、API 串接規格

- OpenAI API：文字生成，依上述 prompt 樣板組裝後呼叫
- 呼叫成功後，將回傳內容與 metadata（生成時間戳、使用的 prompt 版本）一併寫入 Notion（透過 Notion API）
- 每筆生成需記錄完整的輸入參數快照（該次呼叫實際使用的角色設定、時間點、事件描述），以利之後追溯每一筆內容的生成依據
